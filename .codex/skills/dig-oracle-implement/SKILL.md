# Dig Oracle Implement

`dig` で実装前提を深掘りし、repoから必要最小限の文脈を加えた `context pack` をOracleへ渡して、実装方針、コード案、テスト案、注意点を得るためのSkillです。Oracleは直接コミットする実装者ではなく、文脈パック付きの実装案担当として扱います。

## 使うべき状況

- ユーザーが実装前に深掘りし、その内容を使ってOracle Proなどに実装案を出させたい。
- 局所的な機能追加、修正、テスト追加について、外部モデルのコード案や設計案を比較材料にしたい。
- Codexがrepoの実態を読んだ上で、Oracleに渡す文脈を安全に絞りたい。

## 期待する入力

- 実装で達成したい目的。
- 触ってよいファイル、触らないファイル、完了条件。
- 既存仕様、関連テスト、検証コマンド。
- Oracleに外部送信してよい範囲。

## 実行手順

1. まず `dig` と同じ姿勢で、実装目的、成功条件、制約、触ってよい範囲、検証条件を一問ずつ深掘りします。
2. 調べれば分かるrepo事実は、README、AGENTS、関連コード、既存テストから確認します。
3. Oracleへ渡す前に、必要最小限の `context pack` を作ります。
4. `context pack` には次だけを入れます。
   - 実装目的
   - 完了条件
   - 触ってよい範囲
   - 触らない範囲
   - 関連する短いコード抜粋または仕様抜粋
   - 既存の設計・命名・テストの流儀
   - 検証コマンド
   - Oracleに求める成果物
5. 秘密情報、個人情報、認証情報、広いrepo全体、未整理のログ、未コミット差分の不要部分は入れません。
6. Oracle実行が必要な場合は `oracle-consult` を使い、送信前に必ずrenderまたはdry-runで内容を確認します。
7. Oracle出力はそのまま貼らず、Codexがrepoの流儀、既存差分、テスト結果に合わせて採否判断します。

## Oracleへの依頼型

Oracleには、次の観点で短く依頼します。

```text
このcontext packは、実装前にdigで深掘りし、Codexがrepo文脈を最小抜粋したものです。
直接適用するパッチではなく、実装方針、コード案、テスト案、注意点を返してください。
既存の制約と検証条件から外れる提案は避け、迷う点は明示してください。
```

## 出力形式

- 深掘りで決まった実装前提。
- Oracleに渡した `context pack` の範囲。
- Oracle回答の要点。
- 採用する実装案、採用しない実装案、追加で確認する点。
- Codexが実装する場合の次アクションと検証コマンド。

## 禁止事項と確認が必要な操作

- Oracle出力をそのままrepoへ貼り付けません。
- Oracleに生repo、秘密情報、認証済みセッション内容、個人情報、未整理のログを送りません。
- 認証、DB、本番操作、外部API、課金、公開設定を含む実装は、Oracleへの丸投げ対象にしません。
- ネットワーク、第三者npm、ブラウザ操作、外部送信が必要な場合は `oracle-consult` の安全手順に従います。
