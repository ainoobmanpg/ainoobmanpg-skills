# Dig Oracle Spec

`dig` で深掘りした仕様文脈を、Oracleに安全に渡せる `context pack` へ圧縮し、仕様の穴、矛盾、反例、代替案を得るためのSkillです。Oracleは最終判断者ではなく、深掘り済み文脈に対する外部レビュアーとして扱います。

## 使うべき状況

- ユーザーが仕様、方針、調査条件、判断材料を深掘りしてから、頭の良い外部モデルにレビューさせたい。
- 壁打ち後に、仕様の抜け、曖昧さ、反例、代替案を洗い出したい。
- 実装前に、何を作るべきか、何を調べるべきか、どの制約が重要かを固めたい。

## 期待する入力

- 仕様化したい目的、背景、対象読者または利用者。
- 既に決まっている制約、成功条件、失敗条件。
- 未決事項、迷っている選択肢、避けたいこと。
- Oracleに外部送信してよい範囲。

## 実行手順

1. まず `dig` と同じ姿勢で、目的、制約、成功条件、未決事項を一問ずつ深掘りします。調べれば分かるrepo事実は先に読み、ユーザーに聞きません。
2. 新しい洞察が薄くなったら、Oracleへ渡す `context pack` を作ります。
3. `context pack` には次だけを入れます。
   - 目的
   - 決まったこと
   - 制約
   - 成功条件と失敗条件
   - 未決事項
   - 比較中の選択肢
   - Oracleに聞きたい質問
4. 秘密情報、個人情報、認証情報、ホスト固有の詳細、未整理のログ、広すぎるrepo抜粋は入れません。
5. Oracle実行が必要な場合は `oracle-consult` を使い、送信前に必ずrenderまたはdry-runで内容を確認します。
6. Oracle回答をそのまま決定事項にせず、ユーザーと共有して採用、不採用、保留に分けます。

## Oracleへの依頼型

Oracleには、次の観点で短く依頼します。

```text
このcontext packは、仕様作成前の深掘り結果です。
仕様の穴、矛盾、曖昧な前提、反例、代替案、次に確認すべき質問を挙げてください。
最終判断ではなく、Codexとユーザーが仕様を固めるためのレビュー材料として返してください。
```

## 出力形式

- 深掘りで決まったこと。
- 未決事項。
- Oracleに渡した `context pack` の範囲。
- Oracle回答の要点。
- 採用する指摘、保留する指摘、採用しない指摘。

## 禁止事項と確認が必要な操作

- このSkillでは実装、ファイル編集、Git操作、Issue化、PR化を行いません。
- Oracleへ生repo、秘密情報、認証済みセッション内容、個人情報、未整理のログを送りません。
- Oracle回答をユーザー合意なしに仕様決定として扱いません。
- ネットワーク、第三者npm、ブラウザ操作、外部送信が必要な場合は `oracle-consult` の安全手順に従います。
