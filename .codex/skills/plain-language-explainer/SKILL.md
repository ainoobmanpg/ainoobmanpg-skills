# Plain Language Explainer

Codexの進捗、完了報告、失敗説明、技術判断を、非技術者が判断できる日本語へ言い換えるためのSkillです。技術的な正確さは落とさず、読者が「何が起きたか」「どう判断すればよいか」を先に理解できる形にします。

## 使うべき状況

- ユーザーが、説明をわかりやすく、非技術者向けに、専門用語なしで、簡単にしてほしいと明示した。
- Codexの直前の説明、進捗報告、完了報告、エラー説明を読みやすく言い換えたい。
- 技術的な詳細よりも、結論、確認結果、次に必要なことを優先して伝えたい。

## 期待する入力

- 言い換える対象の説明、報告、ログ要約、エラー内容。
- 何を判断したいのか、何が不安なのか、どこが難しかったのか。
- 実行済みの操作、確認済みの結果、まだ未確認のこと。

入力が不足していても、確認済みの事実と推測を分けて説明します。事実確認が必要な場合は、断定せず「まだ確認できていない」と書きます。

## 実行手順

1. 最初に結論を一文で書きます。
2. 必要に応じて、`結論`、`何をしたか`、`確認できたこと`、`次に必要なこと` に分けます。
3. 専門用語は原則使いません。必要な場合だけ、`CI（自動チェック）` のように直後へ短い説明を添えます。
4. 失敗、危険、未確認のことは、やさしい言葉でもはっきり書きます。
5. コマンド名、ファイル名、ログは必要最小限だけ残し、判断に必要な意味を先に説明します。
6. 技術者向けの詳細が必要な場合は、本文の最後に短く分けて置きます。

## 出力形式

必要に応じて、次の見出しを使います。不要な見出しは省きます。

- 結論
- 何をしたか
- 確認できたこと
- 次に必要なこと

## 禁止事項と確認が必要な操作

- ユーザーが頼んでいないのに、通常の全回答へ常時適用しません。
- 専門用語を並べただけの説明にしません。
- 長いログやコマンド出力をそのまま貼りません。
- 失敗、危険、未確認事項を安心できる表現でぼかしません。
- 比喩で正確さをごまかしません。
- 確認していないことを「確認済み」と書きません。
- このSkill自体では、ファイル編集、コマンド実行、ネットワークアクセス、Git操作、外部送信をしません。根拠確認や追加作業が必要な場合は、別作業として扱います。
