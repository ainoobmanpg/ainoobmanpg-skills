# Away Mode Handoff

ユーザーがしばらく見られない間に、Codexが安全に進められる範囲を明確にするためのSkillです。`goal-setter` は「何がDoneか」を固め、このSkillは「不在中にどこまで任せてよいか」を固めます。

## 使うべき状況

- ユーザーが「これから寝る」「外出する」「しばらく離席」「放置しておく」「戻るまで自走して」と言っている。
- ユーザーが「重要な設計判断は後回し」「自分で決められる範囲で進めて」といった委任境界を示している。
- ユーザーが `goal-setter`、`/goal`、長時間実行、留守番実行と組み合わせたいと言っている。

通常の短い修正、単なる質問、壁打ちだけの依頼では使いません。

## 期待する入力

足りない情報は安全側に倒します。次が分かるほどよいです。

- 不在時間または戻る目安
- 達成したい最終状態
- 許可する操作
- 禁止する操作
- Doneの証拠
- 戻ってきたユーザーに見せたい報告形式

危険操作の境界が曖昧なら、質問せず許可されていない扱いにします。

## 手順

1. ユーザーの目的、不在時間、任せたい範囲を短く復元します。
2. Doneや検証が曖昧なら `goal-setter` を使って、目的、証拠、制約、停止条件を先に固めます。
3. 次の3区分に分けます。
   - 自走OK: ローカル調査、読取、限定編集、通常のテストなど、戻せる範囲。
   - 要確認: 設計方針、公開挙動、認証、課金、個人情報、本番、外部API、削除、移行、push、deploy、購入。
   - 停止: 必須権限、認証、外部判断、危険操作が必要で代替検証できない状態。
4. 自走OKの範囲だけ進めます。低リスクな仮定は明記して続行します。
5. 報告は結果、変更点、検証、残った判断だけに絞ります。

## 委任文テンプレート

ユーザーが文面を欲しがっている場合は、次の形で短く出します。

```text
これから<時間>ほど不在なので、<目的>を進めてください。
許可: <読取、ローカル編集、テスト、調査など>
禁止: <push、deploy、本番変更、削除、課金、認証変更、公開設定変更など>
判断: 安全で戻せる範囲はあなたの判断で進め、重要な設計判断や危険操作は止めてください。
Done: <確認すべき証拠>
停止: <権限不足、検証不能、危険操作が必要、前提が崩れた場合>
報告: 完了、未完了、検証結果、残った判断だけを簡潔にまとめてください。
```

## `/goal` と組み合わせる場合

ユーザーが明示的に `/goal` 化や永続ゴール化を求めた場合だけ、`goal-setter` でコンパクトなGoalにします。Goalには次を必ず含めます。

- 不在中に進める最終状態
- Doneの証拠
- 許可する操作
- 禁止する操作
- 停止条件
- 最終報告で示す項目

`GOAL.md` や長い作業計画ファイルは作りません。実行中のGoalまたは会話内の委任文を正とします。

## 禁止事項

- 「自分で決められる範囲」を、push、deploy、本番変更、購入、削除、公開設定変更、認証変更の許可として扱わない。
- 重要な設計判断が必要な作業を、ユーザー不在中に勝手に確定しない。
- 検証していない作業を完了扱いにしない。
- 代替物、モック、近い例だけでDoneを満たしたことにしない。
- 長いチェックリストや過剰な管理ファイルを作らない。
