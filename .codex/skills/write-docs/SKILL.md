# Write Docs

草案メモ、会話内で整理した内容、既存仕様、運用上の知見をもとに、README、docs配下、仕様メモ、運用メモなどのMarkdown文書を作成または更新するためのSkillです。通常責務はリポジトリ内のMarkdown文書の編集までとし、Issue本文、PR本文、作業票、外部サービスへの送信は別Skillまたは別作業に切り分けます。

## 使うべき状況

- READMEに現状仕様、使い方、検証手順、運用メモを追記または整理する。
- docs配下に仕様、設計判断、運用手順、調査結果をMarkdown文書として残す。
- 会話内の整理内容や草案メモを、後から読める文書へ整える。
- `thinking-partner` が出した草案メモや論点メモを、ユーザーの明示指示後に文書化する。
- 既存Markdownの構成、見出し、表現、重複を整理し、読み手に合わせて更新する。

## 期待する入力

入力が不足していても、リポジトリ内のREADME、AGENTS.md、関連docs、既存差分から安全に補える範囲で補完します。目的、読者、保存先、編集範囲を推測すると危険な場合だけ、最小限の質問をします。

- 文書化したい目的、背景、完了条件。
- 想定読者、読み手が次に行う行動。
- 保存先候補、新規作成か既存文書更新かの希望。
- 参照する既存文書、仕様、コード、会話メモ、草案メモ。
- 触ってよいファイル、触らないファイル、文体や粒度の制約。
- 未確定事項、決定済み事項、仮説として扱うべき事項。

## 実行手順

1. まずREADME、AGENTS.md、関連docs、対象Markdownを読み、既存の文体、見出し粒度、配置規約を確認します。
2. 文書の目的、想定読者、読み終わった後に期待する行動を固定します。
3. 既存文書への追記、既存文書の整理、新規Markdown作成のどれが最も自然かを判断します。
4. 未確定事項、仮説、決定事項を混ぜず、必要なら「未確認」「未決」「前提」として明示します。
5. 既存のトーン、用語、見出し構造に合わせ、Markdownとして読みやすい構成に整えます。
6. コード、コマンド、パス、設定名は、必要に応じてインラインコードまたはコードブロックで示します。
7. 変更後は、対象ファイルの目視確認、`find . -maxdepth 4 -type f` による配置確認、可能なら `git diff --check` を実行します。

## `thinking-partner` からの引き継ぎ

`thinking-partner` から引き継ぐ場合は、次のような草案メモを入力として受け取れます。すべての項目が埋まっている必要はありませんが、空欄や未確定の内容を決定事項として扱いません。

```markdown
## 背景

<なぜ文書化するか。>

## 目的

<文書で達成したいこと。>

## 決まったこと

- <決定事項>

## 未決事項

- <まだ判断が必要なこと>

## 選択肢とトレードオフ

- <選択肢>: <利点、懸念、向いている条件>

## 次のアクション

- <文書化、確認、別Skillまたは別作業に渡す内容>
```

## 出力形式

作業後は、必要に応じて次の項目を短く報告します。

- 変更対象: 作成または更新したMarkdownファイル。
- 要約: 文書に反映した主な内容。
- 未確認事項: 仕様未確定、情報不足、ユーザー確認が必要な点。
- 検証結果: 目視確認、配置確認、`git diff --check` などの結果。

## 禁止事項と確認が必要な操作

- ユーザーが明示していないのに、Issue化、PR化、作業票化、外部サービス送信へ進めません。
- このSkillの通常範囲では、Git操作、push、リモート作成、公開設定変更を行いません。
- 秘密情報、APIキー、個人情報、認証済みセッション内容を文書に含めません。
- 未確定の仕様、プロダクト判断、運用判断を、決定事項として書きません。
- 既存の文体、配置規約、責務境界を無視して、関係ない文書やコードへ広げません。
- ネットワーク、認証、外部ツール、本番操作、破壊的操作が必要になった場合は、別作業として明示確認してから扱います。
