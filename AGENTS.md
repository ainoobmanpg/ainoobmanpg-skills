# AGENTS.md

このリポジトリは、日本語でCodex用およびClaude Code用のSkillを整備するための作業場です。

## 基本方針

- ユーザー向けの説明、進捗、レビュー、Skill本文は日本語で書く。
- ファイル名、ディレクトリ名、コード識別子は原則として英語またはASCIIのkebab-caseにする。
- Skillは小さく、単独で理解でき、呼び出し条件が明確な単位に分ける。
- Codex向けとClaude Code向けを両方用意する場合、同じSkill名と同じ目的にそろえる。
- 秘密情報、APIキー、個人情報、認証済みセッションの内容をコミット対象に含めない。
- コミットメッセージ、PR本文、レビューコメント、作業メモなど、人が読む文言は基本的に日本語で書く。
- GitHub上ではPrivate Repoとして扱う前提で作業する。公開前提のURL、宣伝文、外部公開用の説明を勝手に追加しない。

## ディレクトリ規約

```text
.codex/skills/<skill-name>/
  SKILL.md
  scripts/
  templates/

.claude/skills/<skill-name>/
  SKILL.md
  scripts/
  templates/
```

`scripts/` と `templates/` は必要な場合だけ作成します。空の補助ディレクトリを維持したい場合は `.gitkeep` を置きます。

## SKILL.md 規約

各 `SKILL.md` はYAMLフロントマターとMarkdown本文で構成します。

```markdown
---
name: skill-name
description: "このSkillが必要になる依頼、対象、ユーザーが言いそうな語彙を一文で説明する。"
---

# Skill Name

このSkillが何をするか、いつ使うか、何を禁止するかを具体的に書く。
```

本文には最低限、次を含めます：

- 使うべき状況
- 入力として期待する情報
- 実行手順または判断基準
- 出力形式
- 禁止事項、確認が必要な操作、ネットワークや外部ツールの前提

`description` はSkill本文が読まれる前の主要な発火条件です。単なる概要ではなく、いつ使うか、対象リソース、ユーザーが使いそうな言い回し、近いSkillとの切り分けを短く含めます。コロン、バッククォート、記号、箇条書き風の表現を含める場合は、YAMLの型事故を避けるため引用符で囲みます。

## Codexで作業するとき

- まず既存のSkill構成とREADMEを確認してから変更する。
- 既存ファイルを編集する場合、関係ない差分を戻さない。
- 手作業の編集には `apply_patch` を使う。
- 変更後は可能な範囲で `find . -maxdepth 4 -type f` やMarkdown本文の目視確認を行う。
- Git操作、ネットワーク、破壊的操作、外部認証が必要な作業は事前に明示する。
- コミットするときは、日本語の件名と本文にする。例: `初期構成を追加`。
- リモート作成、push、公開設定の変更は、ユーザーから明示された場合だけ行う。

## Claude Codeとの互換性

- Claude Code向けの追加制約は `CLAUDE.md` と対象Skillの本文に書く。
- Codex固有の運用をClaude Code側に押し付けない。
- 共通化できる説明は同じ文面に寄せ、ツール差分だけを分ける。

## レビュー観点

- 呼び出し条件が広すぎないか。
- `description` だけを読んでも、エージェントが使うべき依頼と使わない依頼を判断できるか。
- エージェントが勝手に危険な操作をしない設計になっているか。
- 実行手順が曖昧すぎず、過度に長すぎないか。
- 日本語の説明が自然で、ユーザーがそのまま読んでも理解できるか。
- Codex版とClaude Code版の差分に理由があるか。
